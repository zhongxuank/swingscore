# SwingScore Agent Handoff

## Read This First
- For the full current status, completed-change summary, limitations, and next-build roadmap, read `docs/BETA_HANDOFF.md`.
- This file is the quick operational handoff; keep it concise and update `docs/BETA_HANDOFF.md` when project direction or implementation scope changes.

## Current Product Direction
- The app is moving from an alpha demo into a Beta model built around **Contests with Rounds**.
- Use **Contest** for the division-level container, such as `Novice Jack & Jill`.
- Use **Callback Round** for individually scored, role-based rounds such as Prelim, Quarter-Final, and Semi-Final.
- Use **Final Round** for couple-scored Relative Placement rounds.
- Chief Judge scoring is always raw-score entry. If Chief Judge scores count in a final, include them in the odd scoring-judge count; otherwise exclude them.
- Configure Chief Judge score-counting from the Judges tab. For Callback Rounds, the Chief Judge can count for Leaders, Followers, or Both depending on their assignment.

## Local Alpha State
- Contest Manager create/edit/archive/restore is localStorage-backed, not Supabase-backed.
- Local contests and rounds are stored through `src/lib/data/local-contest-store.ts`.
- The demo contest still provides the most complete scoring data. Newly-created contests can create rounds and access links, but their competitor, score, heat, pairing, and submission data is not fully persisted yet.
- Judge, Chief Judge, and Emcee routes now understand generated local access-link tokens enough to show the correct contest and round header context.
- The scoring bodies of `/judge/[token]`, `/chief/[token]`, and `/emcee/[token]` are still demo-data-backed until a real round data loader is implemented.

## Access Link Slice
- Round setup now includes an **Access** tab for Callback and Final rounds.
- Access links are generated by `src/lib/access-links.ts`.
- Access links are generated only after the round judge panel is confirmed.
- Judge links include names and role labels. The Access tab also provides a WhatsApp-friendly copy block for sending judge-sheet links.
- Local token format is deterministic and readable for alpha QA only. Do not treat it as a secure token model.
- Supabase Beta should replace local deterministic tokens with stored opaque tokens, hashed at rest, scoped by round and role.
- Access links should eventually load all round-specific data: contest, round, judge assignment, competitors, heats or pairings, scores, submissions, and result snapshots.

## Recommended Next Implementation Steps
- Persist per-contest competitors, heats, pairings, judge assignments, scores, submissions, and round snapshots beyond the current local contest/round shell.
- Replace demo-data reads in `judge-workspace.tsx`, `chief-workspace.tsx`, and `emcee-workspace.tsx` with a shared access-token data loader.
- Make `/judge/[token]` branch by round scoring method:
  - Callback Round: role-based individual raw scoring.
  - Final Round: couple raw scoring converted to ordinals.
- Make `/chief/[token]` branch by round scoring method:
  - Callback Round: boundary review and Chief Judge raw tiebreak scores.
  - Final Round: panel status, raw-score review, and Relative Placement finalization support.
- Make `/emcee/[token]` read finalized snapshots for callbacks, alternates, finalist pairings, heat sheets, and placements.
- Add Supabase migrations and RLS policies for access links before treating any generated link as production-safe.

## QA Expectations
- After setup UI changes, run:
  - `npm run lint`
  - `npm run typecheck`
  - `npm test`
  - `npm run test:e2e`
- Use the in-app browser to verify create/open/edit/archive/restore and the generated Access tab links.
- Avoid running `npm run build` while the dev server is serving the same `.next` directory unless you plan to restart `npm run dev` afterward.
